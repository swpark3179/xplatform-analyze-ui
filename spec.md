[프로젝트 개요]
특정 프로젝트를 읽어서 참조관계를 분석하는 프로그램이야.
tauri 프로젝트로 구성되어 있으며, ui는 react 기반으로 그리고 엔진은 rust로 작성되어 있어.
UI 테마는 nord light 테마로 구성하면 돼.

[기능 흐름]
1. 프로젝트 Root 폴더를 선택해
2. Root/src/webapp/ui 폴더 하위에서 존재하는 모든 xfdl 파일 각각을 읽어서 목록화 시켜줘. 목록화 도중에는 프로그레스 바 표시가 필요해. 이 목록에서 분석대상을 선택해서 체크박스 on을 시키는데, 검색어 입력을 통해 필터링을 해가며 분석대상 목록을 완성해.
3. 분석시작 버튼을 클릭하면 본격적으로 분석을 시작할거야.
 - xfdl 파일은 xml 영역과 Script 영역으로 구성되어 있어.
 - 기본적으로 파일 전체는 xml 형태로 구성되어 있고, <Script> 태그 내에 <![CDATA[ ~~~ ]]> 블록으로 감싸져 있는 내부는 javascript와 유사한 문법으로 구성되어 있어.
 - 이 스크립트 내부에서 actionSubmit 또는 acationSubmitFR 함수를 호출하는 모든 위치에 대해서, 묻지도 따지지도 말고 분석결과에 포함되어야 해. (서비스를 찾으면 성공, 못찾으면 실패로 리턴될거야)
 - 각 actionSubmit 또는 actionSubmitFR 호출하는 부분이 액션 call 하는 부분이고, 첫 번째 파라미터가 actionId야. 유일값으로 처리하기 위해 (확장자를 제거한 파일명).action(첫번째 파라미터) 를 분석결과 Id로 취급하자.
4. 분석이 모두 끝나면 분석결과 Id별로 가상 테이블 형태로 표현될거야. 각 아이템을 클릭하면 세부 정보가 보일거야.
5. 파일 저장 버튼을 클릭하면 결과를 엑셀파일로 저장할 수 있어.

[분석 세부로직]
1. 각 분석Id에 대해 Java 서비스 위치를 찾아야 해.
- 서비스 위치는 xfdl 파일 내의 태그들 중에서 <Dataset> 태그를 모두 찾는다. 이 태그들 중 id 속성값이 "dsAction" 인 것을 찾으면 xfdl 파일 내에서 하나만 존재한다.
- 이 dsAction 데이터셋 내에서 모든 <Row> 태그를 수집한다. 각각의 Row 태그마다 하위에 <Col> 태그가 있고, 이 <Col> 태그들 중 id 속성값이 ID 인 것과 URL 인 것을 쌍으로 가지고 있으면 된다. 이 때 ID인 것의 내부텍스트를 각 분석Id의 actionId와 비교해서 일치하는 것의 URL이 실제 Java 서비스가 되는 경로이다. 여기까지는 예외없이 수집하도록 하자.
2. src/webapp/ui/default_typedef.xml 파일을 분석하자.
- 이 파일이 없으면 분석 실패를 띄우고, 무조건 이 경로에 파일은 존재해야 한다.
- 이 파일에는 xml 기반으로 구성되어 있다.
- <Service> 태그를 모두 수집한다. 여러 속성들이 포함되어 있을텐데, url 속성이 "./" 로 시작하는 모든 서비스들을 수집한다. (수집대상은 prefixid 속성값과 url 속성값
- 1에서 찾아낸 URL의 처음 부분 (처음으로 슬래시(/) 가 등장하기 전까지) 를 수집된 서비스에서 prefixid와 일치하는 것을 찾는다. 못찾으면 수동분석 대상으로 분류하고 분석을 종료한다 (다른 Action 분석으로 넘어간다)
- 프로젝트의 src/java/com/shi/ 경로에서부터 찾은 서비스의 url과 같은 경로로 진입한다. 대소문자 관계없이 경로를 찾는다.
- (1에서 찾아낸 URL의 두번째 부분 - 처음 슬래시 이후부터 두번째 슬래시 앞까지) 부분을 {파일ID} 라고 한다면, 찾은 java 경로 내에서 {prefixid를 대문자로 변환한 문자}{파일ID}Service.java 를 찾는다. 예를 들면, actionId가 "/cmcs/0246/selectApprovalParam"이고, default_typedef.xml 파일에서 <Service prefixid="cmcs" type="form" url="./cm/cmc/cmcs" version="0"/> 와 같이 정의되어 있다면, src/java/com/shi/cm/cmc/cmcs/CMCS0246Service.java 파일을 찾아야 한다.
- 그리고 해당 파일 내에서 (1에서 찾아낸 URL의 마지막 부분 - 마지막 슬래시 이후부터) 가 메서드 이름으로 된 메서드를 찾는다. public 으로 선언된 메서드만 대상으로 메서드 명을 찾아서, 해당 블록이 시작되는 라인번호를 찾으면 된다.
3. 이제부터는 바이너리 기반으로 참조관계를 분석하려고 한다. 일단 한단계만 들어가보자. src/java/패키지경로/파일명.java 이 java 파일이고, 맵핑되는 바이너리 파일은 target/classes/패키지경로/파일명.class 를 찾으면 된다. 파일명*.class 가 더 정확할지도 모르겠다.
- 이렇게 찾은 java 파일과 class 파일을 프로그램 내에서 로그로 한번 출력하고, 이 메서드 위치도 객체로 추출하여 정보를 추출하자. 또한, 이 메서드에서 호출하는 모든 참조를 각각 콘솔 로그에 출력하는것까지를 1차 기능 마무리고 구성하자.
